# turn-mcp-web — AI Agent 使用指南

这份文档告诉你（AI Agent）如何正确使用 `turn.wait` MCP 工具。  
读一遍，在整个任务过程中遵守这些规则。

---

## turn.wait 的作用

`turn.wait` 暂停你的执行，将消息发送到人工操作员的浏览器控制台。  
你会阻塞，直到操作员回复为止。收到回复后继续执行。

这是一个**强制性的人在环路检查点**。在执行需要人工审核的操作之前，用它获取审批、确认或指引。

---

## 何时调用 turn.wait

**必须在以下操作前调用：**
- 删除、覆盖或移动文件和数据
- 执行数据库迁移或 Schema 变更
- 部署到任何环境（测试环境或生产环境）
- 进行有副作用的 API 调用（支付、发送邮件、触发 Webhook）
- 任何不可逆或难以撤销的操作
- 代表用户提交并推送代码

**在以下情况下调用：**
- 存在多种可行方案，需要人工选择
- 遇到无法从上下文推断的歧义
- 一个子任务完成，人工需要审查后才能继续下一步
- 即将开始一个较长的自主任务——先确认范围
- 任务完成——汇报结果并询问是否还有其他需求

**不要调用的情况：**
- 你可以从对话或上下文中推断出答案的问题
- 操作可以轻松撤销（如读取文件、分析代码）
- 在同一会话中已经问过相同的问题

## 如何调用 turn.wait

工具名称：`turn.wait`（别名：`turn_wait`、`turn`）

### 参数

| 参数 | 必填 | 说明 |
|---|---|---|
| `context` | **是** | 已完成的内容和即将执行的操作。要具体、简洁。 |
| `question` | 否 | 需要人工回答的具体问题。省略时人工只看到 context。 |
| `options` | 否 | 最多 10 个预设回复选项，以按钮形式显示。选项清晰时（2–5 个）使用。 |
| `agentName` | 否 | 你的名称或子任务名称，帮助人工识别是谁在问。 |
| `timeoutSeconds` | 否 | 等待超时时间（10–3600 秒）。默认由服务器配置决定。 |

### 如何写好 context

`context` 是人工首先看到的内容。写的时候要让人工能在不追问的情况下做出决策。

**好的 context：**
```
代码分析完成。发现 3 个可删除的文件：
- src/legacy/old_auth.py — 最后提交 8 个月前，无任何引用
- tests/skipped_suite.py — 全部 12 个测试已标记 @skip
- scripts/migrate_v1.sh — 迁移脚本，生产环境已于 2024-03-01 执行完毕

即将永久删除这 3 个文件。
```

**不好的 context：**
```
找到了一些可以删除的文件，是否继续？
```

### 使用 options

当有明确的少量选项时使用：

```json
{
  "context": "...",
  "question": "应该先部署到哪个环境？",
  "options": ["测试环境", "生产环境", "两者都部署", "取消"]
}
```

对于需要自由输入的开放性问题，不要使用 options。

## 如何处理回复

人工回复后，你收到的消息末尾会附带一个强化提示后缀。  
该后缀提醒你在下一个检查点再次调用 `turn.wait`——请遵守。

- 人工说**继续 / 是 / 确认** → 执行操作
- 人工说**取消 / 停止 / 否** → 中止并说明未执行的内容
- 人工提供指示或更正 → 将其纳入并继续
- 收到 `[timeout]` → 人工未响应；不要执行不可逆操作，汇报超时并等待下一条指令
- 收到 `[canceled]` → 人工已取消；停止当前任务

---

## 规则

1. **每个重要决策点一次检查点。** 不要把多个不相关的问题合并进一次调用；每个决策点调用一次，而不是整个任务只调用一次。

2. **行动之后再次调用。** 完成一个步骤后，调用 turn.wait 汇报进度并在继续之前确认下一步。

3. **提供足够的决策信息。** 人工看不到你的推理过程。告诉他们：你发现了什么、即将做什么、风险或影响是什么。

4. **遵守强化提示后缀。** 人工回复后，消息末尾的后缀提醒你继续调用 turn.wait。不要忽略它。

5. **不要静默结束会话。** 任务完成时，最后调用一次 turn.wait 汇报结果并询问是否还有需求。

6. **超时后不要自动重试。** 如果 turn.wait 超时，不要自动重试。汇报超时并暂停。

---

## 反模式

| 模式 | 问题 | 修正方式 |
|---|---|---|
| 只在开始时调用一次，之后不再调用 | 第一个检查点之后人工失去可见性 | 在每个主要步骤前后都调用 |
| 没有上下文直接问"是否继续？" | 人工无法做出有依据的决策 | 明确描述即将做什么以及原因 |
| 每个小操作都调用 turn.wait | 频繁打断人工 | 合并相关操作；只在关键步骤前检查点 |
| 超时后继续执行 | 未经人工审批擅自行动 | 暂停并通知 |
| 任务结束时不做最终汇报 | 人工不知道做了什么 | 始终以总结性调用收尾 |

## 最简示例

```python
# 展示正确检查点用法的伪代码

result = analyze_codebase()

# 检查点 1：执行操作前
reply = turn.wait(
    context=f"分析完成。发现 {len(result.deletable)} 个可安全删除的文件：\n{result.summary}",
    question="确认删除？",
    options=["是，删除", "否，跳过", "先给我看完整列表"],
    agent_name="CleanupAgent"
)

if "是" in reply or "删除" in reply:
    deleted = delete_files(result.deletable)
    
    # 检查点 2：操作完成后，结束前
    turn.wait(
        context=f"已删除 {len(deleted)} 个文件，释放 {deleted.size_mb:.1f} MB。\n所有文件在回收站中，30 天内可恢复。",
        question="任务完成。还有其他需求吗？",
        agent_name="CleanupAgent"
    )
```

## 总结

- `turn.wait` 是你在任务过程中与人工沟通的方式
- 在**不可逆操作前**、**完成步骤后**、**任务结束时**都要调用
- 写 `context` 时假设人工没有任何其他信息——通常确实如此
- 遵守每次回复中的强化提示后缀
- 有疑虑时，设置检查点
